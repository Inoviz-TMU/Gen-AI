Storing and Retrieving Knowledge at Scale

What is a Vector Embedding?
A vector embedding is a list of numbers (e.g., 1536 decimal values) that represents the meaning of a piece of text. Texts with similar meanings produce vectors that are close together in this number space. Think of it as coordinates on a map — "dog" and "puppy" land near each other, while "dog" and "stock market" land far apart.
Analogy: Imagine a library where every book is placed not by author or title, but by what it's about. Books about similar topics end up on the same shelf. Vector search lets you walk to the right shelf, even if you use completely different words than the book's title.

Keyword Search (BM25)

Query: "time off policy"
Document says: "employees are entitled to 20 days of paid annual leave"

No match. The words don't overlap, so traditional search fails to find it.

Vector Search

Query: "time off policy"
Document: "employees are entitled to 20 days of paid annual leave"

Similarity: 0.89 — The embeddings understand "time off" ≈ "paid annual leave" in meaning.

Indexing Strategies

What is an Index?
An index is a data structure built over your vectors that makes search fast. Without an index, finding the closest vector to your query means comparing it against every single vector — like looking for a word in a book by reading every page. An index organizes vectors so you can skip most of them and find the answer in milliseconds.

The index type determines the fundamental trade-off between query speed, accuracy (recall), and memory usage.

Flat Index (Brute Force)

Exact Match
Definition
No data structure at all — just your raw vectors stored in a list. When a query comes in, the system calculates the distance (similarity) between the query and every single vector, one by one, then returns the closest ones. It's like scanning every item in a supermarket to find the cheapest milk.
Analogy: You have 10,000 photos on your phone. To find one of a sunset, you swipe through every single photo. Slow, but you'll definitely find it if it exists. No photo gets skipped.
100% recall — guaranteed to find the true closest vectors, no approximation
Zero build time — no index to construct, just store vectors
Low memory — only stores the vectors themselves, no extra structure
O(n) latency — gets linearly slower as you add more vectors. 1M vectors? Every query checks all 1M.
Not viable at scale — unusable beyond ~100K vectors for real-time queries
Use when: Dataset is small (<100K vectors), or you need a baseline for benchmarking other indexes against.

IVF (Inverted File Index)

Partitioned
Definition
During index build, runs k-means clustering to group your vectors into, say, 256 clusters (partitions). At query time, instead of checking all vectors, it first finds the closest few cluster centers, then only searches within those clusters. The nprobe parameter controls how many clusters to check.
Analogy: A supermarket organizes products into aisles (clusters). Instead of checking every shelf, you first figure out "milk is in aisle 3 and 7" (find closest cluster centers), then only walk down those 2 aisles. Much faster, but you might miss milk if it was accidentally placed in aisle 9.
Much faster than flat — only searches a fraction of vectors
Tunable accuracy — increase nprobe for better recall, decrease for speed
Good for 1M-100M vectors — practical sweet spot for many use cases
Requires training — must run k-means on a sample before indexing
Can miss results — if the closest vector is in a cluster you didn't check
Not great for real-time inserts — adding vectors to existing clusters is expensive
The nprobe knob: With 256 clusters, nprobe=8 means you search 8/256 = 3% of vectors. nprobe=64 searches 25%. nprobe=256 = flat search (100%). Typical: nprobe=8 to 32.
Use when: Dataset is 1M-100M vectors, you need a balance of speed and accuracy, and your data is relatively static (batch updates, not streaming).

HNSW (Hierarchical Navigable Small World)

Graph-Based — Industry Default
Definition
Builds a multi-layer graph where each vector is a node connected to its neighbors. The top layer has few long-range connections (like highways between cities). The bottom layer has all vectors with many short-range connections (local roads). Search starts at the top layer, greedily hops toward the query, then drops down a layer to refine. Like zooming in on Google Maps — start with country-level highways, end at street-level roads.
Analogy: Think of a video game minimap. The world map (top layer) shows you which region to go to. The region map (middle layer) shows which neighborhood. The street map (bottom layer) shows the exact house. You navigate down layers, getting more precise at each step — never needing to check every house.
O(log n) speed — extremely fast, barely slows down as dataset grows
High recall (95-99%+) — very few misses in practice
Supports incremental inserts — add vectors without rebuilding
Default in Pinecone, Weaviate, Qdrant — battle-tested
High memory usage — storing all those graph edges can use 2-4x the raw vector memory
Slow index build — constructing the graph is computationally expensive
Tuning knobs are non-obvious — M, efConstruction, efSearch interact in complex ways
Key parameters: M = max connections per node (default 16, higher = better recall but more memory). efConstruction = build-time accuracy (default 200, higher = better index but slower build). efSearch = query-time beam width (default 50-200, higher = better recall but slower query).
Use when: Production workloads of any size. This is the default choice for most real applications. Use it unless you have a specific reason not to.

Metadata Filtering & Hybrid Search

What is Metadata Filtering?
Each vector can carry key-value tags (metadata) — like a label on a file folder saying "Department: HR", "Type: PDF", "Date: 2024-03". Filtering means only searching within vectors that match your criteria, like telling the librarian "only look in the HR section." This dramatically reduces irrelevant results.
# Only search within HR department PDFs from 2024 index.query( vector=query_emb, top_k=5, filter{"department":{"$eq":"HR"}, "doc_type":{"$in":["pdf","docx"]}, "year":{"$gte":2024}} )
What is Hybrid Search?
Combines vector (dense) search with keyword (sparse) search. Dense search finds meaning; sparse search finds exact words. Hybrid fuses both result sets using Reciprocal Rank Fusion (RRF). This catches cases where exact terms matter (product codes "XR-500") that embeddings might miss.

Chunking Strategies

What is Chunking and Why Does It Matter?
Embedding models have a maximum input length (e.g., 512 or 8192 tokens). You can't stuff an entire 50-page PDF into one embedding. So you split documents into smaller pieces (chunks), embed each chunk separately, and store them individually in the vector DB.

Why it matters: How you split directly affects retrieval quality. A bad chunk cuts a sentence in half — the embedding loses context and the search fails. A good chunk keeps complete thoughts together so the embedding captures full meaning.
Analogy: Imagine cutting a long article into paragraphs before filing them. If you cut mid-sentence, the paragraph makes no sense on its own. If you cut at natural paragraph breaks, each piece is self-contained and understandable. Chunking is the same — cut at the right boundaries.
Isolation

Namespace & Multi-tenancy Patterns

What is Multi-tenancy in Vector DBs?
When multiple customers or teams share the same vector database, you need to isolate their data so Company A's query never sees Company B's documents. Namespaces, partitions, and separate indexes are three ways to achieve this — each with different trade-offs.

Namespace Isolation

ns: acme-corp — 12,400 vectors
ns: globex-inc — 8,200 vectors
ns: wayne-ent — 23,100 vectors

Same index, zero cross-tenant leakage.

Pattern Trade-offs

Namespaces: Easiest ops, shared resources, noisy-neighbor risk.

Separate Indexes: Full isolation, independent scaling, higher ops.

Metadata Filtering: Most flexible, but ensure filters are never bypassed.

Rule: Namespaces for 10-1000 tenants. Separate indexes for 1-10 large tenants.

Operations

CRUD Operations & Index Management

Production vector DB usage goes beyond simple upsert + query. You need idempotent writes, bulk operations, and lifecycle management.

# CREATE — Upsert vectors with metadata index.upsert(vectors=[{"id":"doc1-chunk-0","values":emb,"metadata":{"doc":"handbook.pdf","dept":"HR"}}],namespace="acme")
# READ — Query with filters res=index.query(vector=q,top_k=5,include_metadata=True,filter={"dept":{"$eq":"HR"}},namespace="acme")
# UPDATE — Replace vector + metadata index.update(id="doc1-chunk-0",values=new_emb,metadata={"dept":"People"},namespace="acme")
# DELETE — By ID or by metadata filter index.delete(filter={"doc":{"$eq":"handbook.pdf"}},namespace="acme")

RAG Pipeline Anatomy

What is RAG?
Retrieval-Augmented Generation is a technique that gives an LLM access to external knowledge. Instead of relying only on what the model memorized during training, RAG fetches relevant documents from a knowledge base and includes them in the prompt. This lets the LLM answer questions about your private data — documents it has never seen before.

The canonical pipeline has five stages. Click play to watch each step activate.

Ingest

Load docs from S3, SharePoint. Parse PDF/DOCX. Extract text + metadata.

Embed

Convert chunks to vectors via embedding model (e.g., text-embedding-3-small).

Index

Store vectors + metadata in vector DB. Build HNSW graph. Assign namespace.

Retrieve

Embed query, search vector DB, apply filters, re-rank with cross-encoder.

Generate

Inject context into LLM prompt. Generate answer with citations. Apply guardrails.

Problem

Naive RAG and Its Limitations

What is Naive RAG?
The simplest possible RAG: chunk document → embed all chunks → for each query, retrieve top-k → stuff them into the LLM prompt → get answer. No query transformation, no re-ranking, no smart chunking. It works for demos but breaks in production.

Here are the four most common failure modes:

Lost in the Middle

LLMs pay less attention to context in the middle of long prompts. Chunks 4-7 of top-10 are effectively invisible.

Mismatched Granularity

A 512-token chunk has the answer but lacks surrounding context. Or the answer spans two chunks and neither alone suffices.

Query-Vocabulary Gap

User asks "how do I take time off?" but docs say "PTO request procedure." Embedding may not bridge this without query rewriting.

Noisy Retrieval

Top-k returns irrelevant chunks that confuse the LLM. Without re-ranking or score thresholds, garbage-in-garbage-out.

Evolution

Advanced RAG Techniques

These techniques fix naive RAG failures by operating on the query and retrieval process before context reaches the LLM.

Query Rewriting

Use a small LLM to rewrite the user's ambiguous query into a specific, self-contained search query.

User: "what's our policy?"
Rewritten: "company remote work and PTO policy for full-time employees"

HyDE

Ask LLM to generate a hypothetical answer, then embed that answer (not the question) to search.

Query: "vacation rules"
HyDE: "Employees receive 20 days paid leave..."

Multi-Query

Generate multiple variant queries, retrieve for each, then union and deduplicate results.

Original: "security practices"
+ "data protection policies"
+ "compliance requirements"
Refinement

Re-ranking with Cross-Encoders

What is Re-ranking?
Your first search (bi-encoder) is fast but imprecise — it embeds query and document separately. A cross-encoder reads the query AND document together as a pair, producing a much more accurate relevance score. But it's too slow to run on millions of documents. Solution: use bi-encoder to get top-50 candidates, then cross-encoder to re-rank those 50 down to top-5.
Analogy: Bi-encoder = a recruiter skimming 1000 resumes in 5 minutes (fast, rough). Cross-encoder = the hiring manager doing 30-minute deep interviews with the top 10 candidates (slow, precise). You need both.

Parent-Document & Hierarchical Retrieval

What is Parent-Document Retrieval?
Store small chunks for precise search (so the embedding is focused), but keep a reference to the larger parent document. When a small chunk matches your query, instead of sending just that tiny chunk to the LLM, you expand it to the full parent for rich context. This solves the granularity problem — you get precise matching and full context.
Analogy: A book's index says "remote work — see page 42." The index entry (small chunk) is precise enough to find the right page, but you read the entire page (parent document) for full understanding, not just the index line.
Parent Document

"Employee Benefits & Policies — covers health insurance, retirement plans, paid time off, remote work arrangements, and professional development..."

Child Chunks (indexed in vector DB)
Chunk 0: "All full-time employees are entitled to 20 days of paid annual leave..."
Chunk 1: "Remote work policy allows up to 3 days/week..." MATCHED
Chunk 2: "Health insurance through BlueCross includes medical, dental..."
Result: Chunk 1 matched → expanded to full parent → LLM receives rich context about all benefits, not just the remote work sentence.
Frontier

Agentic RAG

What is Agentic RAG?
Instead of retrieval being a fixed preprocessing step that always runs before the LLM, the LLM agent decides when, whether, and how to retrieve. It has retrieval as a tool it can call — just like it might call a calculator or web search. The agent can query multiple sources, decompose complex questions, evaluate results, and retry if the first retrieval wasn't good enough.
Analogy: Standard RAG = a student who always checks the textbook before answering every question, even "what is 2+2?". Agentic RAG = a student who decides when to check the textbook. Simple math? No need. Complex policy question? Check the book, evaluate if the answer is sufficient, maybe check a second source.
