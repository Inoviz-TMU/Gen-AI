Building Agentic Systems: Fundamentals, Memory, and Frameworks

Definitions and working notes on how agents perceive, reason, act, and remember — and how frameworks like LangChain and LangGraph turn these ideas into production systems.

4.1

Agent Fundamentals

What makes an LLM call an "agent" — the control loop, the moving parts, and the trade-offs that show up once you move from a demo to a production system.

Concept

The Observe–Think–Act loop and its production implications

An agent is a system that repeatedly observes its environment, reasons about what to do next, and takes an action — looping until a goal is reached or a stop condition fires.

This is the minimal definition of "agent" as distinct from a single LLM call: a plain prompt-response is one shot; an agent keeps going, feeding the result of its own actions back into its next decision.

Observe→ Think→ Act→ (feed result back)

Production implications: each loop iteration costs tokens and latency, so unbounded loops are a real cost and safety risk. Production agents need a max-iteration cap, a timeout, explicit success/failure detection, and logging of every step for debugging — none of which matter in a notebook demo but all of which matter the moment real users or real money are involved.

Concept

Agent anatomy: perception, reasoning, action, memory

Every agent, regardless of framework, can be decomposed into four functional components.

ComponentRole
PerceptionTurns raw environment state — user messages, tool outputs, documents, sensor data — into a form the model can reason over.
ReasoningThe LLM call itself: interprets the current state and decides the next step (which tool, what answer, whether to stop).
ActionExecutes the decision — calling a tool, writing to a file, sending a message, querying an API.
MemoryCarries information across loop iterations and across sessions, so the agent isn't reasoning from a blank slate each turn.
Pattern

Observe–Think–Act (ReAct) loop in detail

ReAct interleaves natural-language reasoning ("thought") with concrete actions ("tool calls"), so the model explains its plan before it acts and can revise the plan based on what comes back.

A typical ReAct step looks like: Thought → Action → Observation → Thought → .... Making the reasoning explicit (rather than hidden) improves tool-selection accuracy and gives you a human-readable trace of why the agent did what it did — critical for debugging in production.

Pattern

Plan-and-Execute — when to plan upfront vs dynamically

Plan-and-Execute separates planning (produce a full multi-step plan upfront) from execution (carry out each step, optionally replanning if a step fails).

  • Plan upfront when the task is well-scoped, steps are largely independent, and you want predictable cost/latency (e.g. "research these 5 companies and summarise each").
  • Plan dynamically (ReAct-style) when each step's outcome genuinely changes what should happen next — exploratory debugging, open-ended research, conversations.
  • Hybrid: plan upfront, but allow a replanning step whenever execution deviates from expectation.
Pattern

Reflexion — agents that learn from their own failures within a session

Reflexion has an agent critique its own failed attempt in natural language, store that critique as a short-term "lesson," and retry the task with the lesson in context — without any weight updates.

This is learning purely through the context window: the model isn't fine-tuned, but its next attempt is conditioned on an explicit verbal reflection of what went wrong last time (e.g. "the previous SQL query failed because the column name was wrong — check the schema first"). It's cheap and effective for tasks with a fast, checkable feedback signal like code execution or test results.

Constraint

Token budgeting and working memory management

Token budgeting is the practice of actively managing how much of the context window is spent on history, tool outputs, and instructions, so the agent doesn't run out of room or drown in irrelevant context.

  • Truncation — drop or shorten old turns once a limit is hit.
  • Summarisation — periodically compress older history into a shorter summary.
  • Selective inclusion — only pull in the tool outputs/documents relevant to the current step, rather than keeping everything.

Left unmanaged, growing context both increases cost per call and degrades reasoning quality, since relevant details get buried in noise.

Design principle

Agent reliability: when to trust the model, when to add determinism

Reliability engineering for agents means deciding, per decision point, whether to let the LLM decide freely or to constrain it with deterministic code.

Trust the modelAdd determinism
Open-ended reasoning, summarisation, ambiguous natural-language decisions.Anything with a correct answer computable in code — arithmetic, formatting, routing on a known field.
Tasks where several answers are acceptable.Safety-critical or irreversible actions (payments, deletions) — gate with validation or human approval.

In practice this means wrapping the model with schema validation, retries, guardrail checks, and — for high-stakes actions — a human-in-the-loop confirmation step.

Focus: Memory

Why memory matters in agentic systems

Memory is what lets an agent's decisions at step 50 be informed by what happened at step 1 — and, across sessions, by what happened yesterday. Without it, every loop iteration (and every new conversation) starts from zero.

Inside a single run, memory is what makes the Observe–Think–Act loop coherent: the agent needs to remember the original goal, prior tool results, and its own earlier reasoning so it doesn't repeat failed actions or contradict itself. This is usually just the growing conversation history — the model's context window acting as short-term memory.

Across runs, memory is what turns a stateless chatbot into something that improves and personalises over time: recalling a user's preferences, remembering facts learned in a past session, or looking back at how a similar task was solved before (as in Reflexion, but persisted long-term). This is where external storage — vector databases, key-value stores, SQL — comes in, because the context window itself is wiped between sessions and is too small to hold everything anyway.

Practically, memory in an agentic system does three jobs:

  • Continuity — keep the current task's state coherent across many loop iterations (working memory).
  • Recall — retrieve relevant facts, past interactions, or documents on demand, rather than keeping everything in context permanently (external / episodic / semantic memory).
  • Learning without retraining — accumulate lessons, preferences, and corrections in storage the agent reads from, which is far cheaper than fine-tuning and takes effect immediately.

Session 4.2 breaks this down into concrete memory types and how to implement each one.

4.2

Memory Architecture

The concrete storage patterns behind "agent memory" — what to use for short-lived context, what to use for long-term recall, and how to decide when to write and how to retrieve.

Concept

Four memory types and when to use each

Agent memory is usually split into four categories, distinguished by lifespan and structure rather than by implementation.

TypeLifespanTypical use
In-contextOne run / conversationImmediate task state, recent turns, current tool outputs.
ExternalPersistentGeneral long-term storage — documents, facts, arbitrary key-value data.
EpisodicPersistentRecords of specific past agent runs/episodes — "what happened last time."
SemanticPersistentStructured, de-duplicated knowledge — entities, facts, relationships.
Type 1

In-context (working) memory — scope, limits, and management

In-context memory is whatever currently sits inside the model's context window: system prompt, conversation history, and recent tool outputs.

Scope: exists only for the duration of the run; gone once the session ends unless explicitly persisted elsewhere. Limits: bounded by the model's context window and, practically, by the token-budgeting concerns from 4.1 — long-running agents will overflow it. Management: truncate oldest turns, summarise older segments, or keep only the last N tool results in full and compress the rest.

Type 2

External memory — vector stores, key-value, SQL for long-term recall

External memory is any persistent store outside the context window that the agent reads from and writes to via a tool call.

StoreGood for
Vector storeSemantic similarity search over unstructured text — "find things related to X" (Pinecone, Chroma, pgvector).
Key-valueFast exact lookup of known identifiers — user preferences, session state (Redis, DynamoDB).
SQLStructured, relational data with filters/joins — order history, precise records.
Type 3

Episodic memory — storing and retrieving past agent runs

Episodic memory stores whole past episodes — the goal, the sequence of actions taken, and the outcome — so an agent can recall "what happened last time I did something like this."

It's the persisted counterpart to Reflexion: instead of a within-session reflection that's discarded, successful and failed trajectories are logged and can be retrieved (often via vector similarity on the task description) before a new attempt, giving the agent worked examples of its own past behaviour.

Type 4

Semantic memory — structured knowledge bases and entity stores

Semantic memory holds general, de-duplicated facts and relationships — not tied to any one episode — typically as entities and attributes (a knowledge graph or structured profile) rather than raw text.

Example: after ten conversations with a user, episodic memory has ten separate transcripts; semantic memory has the single distilled fact user.dietary_preference = "vegetarian". Semantic memory is what you query when you need the current best-known fact, not the history of how it was learned.

Strategy

Memory write strategies: when to save, what to save, how to summarise

  • When to save — at natural checkpoints: end of a task/episode, when a durable fact is confirmed, or when explicitly asked ("remember that I prefer...").
  • What to save — outcomes and durable facts, not raw intermediate reasoning; save the distilled result, not the whole scratchpad.
  • How to summarise — use a cheap/fast LLM call to compress a transcript into a short structured note before writing, so retrieval later is fast and relevant.
Strategy

Memory retrieval strategies: exact lookup vs semantic vs hybrid

StrategyHow it worksBest for
Exact lookupFetch by known key/ID.Structured facts you already know how to name (user_id → preferences).
Semantic searchEmbed the query, retrieve nearest neighbours by vector similarity.Fuzzy recall — "anything related to this topic" over unstructured memory.
HybridCombine keyword/metadata filters with vector similarity (e.g. filter by user_id, then rank by similarity).Most production systems — narrows the search space before the expensive semantic step.
Worked Example

A small multi-agent system with shared memory (GPT-4o mini)

Two agents collaborate on a research task: a Researcher agent gathers findings and writes them to a shared memory store, and a Writer agent reads that memory to produce a final summary. This shows external memory (a simple key-value dict standing in for a real store) plus a write strategy (summarise-then-save) and a retrieval strategy (exact lookup by topic key).

Agent 1
Researcher
Calls GPT-4o mini to gather raw findings on a topic, then writes a compressed summary into shared memory.
shared store
memory_store
Key-value dict keyed by topic. Stands in for a real external memory (SQL / vector store / KV DB).
Agent 2
Writer
Reads the Researcher's notes from memory (exact lookup by key) and drafts a final answer for the user.
multi_agent_memory.py 
import os
from openai import OpenAI

client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))
MODEL = "gpt-4o-mini"

# ---- shared external memory (stand-in for a real KV / vector store) ----
memory_store = {}

def write_memory(key, content):
    """Write strategy: store a distilled note, keyed for exact lookup later."""
    memory_store[key] = content

def read_memory(key):
    """Retrieval strategy: exact lookup by topic key."""
    return memory_store.get(key, "")


# ---- Agent 1: Researcher ----
def researcher_agent(topic):
    response = client.responses.create(
        model=MODEL,
        instructions=(
            "You are a research agent. Gather 3-4 concise, factual "
            "bullet points on the given topic. Be brief and precise."
        ),
        input=f"Research topic: {topic}",
    )
    findings = response.output_text

    # Summarise-then-save: compress before writing to shared memory
    summary_response = client.responses.create(
        model=MODEL,
        instructions="Compress the following notes into a 2-sentence summary.",
        input=findings,
    )
    write_memory(topic, summary_response.output_text)
    return findings


# ---- Agent 2: Writer ----
def writer_agent(topic):
    notes = read_memory(topic)  # exact lookup, no re-research needed
    if not notes:
        return "No research found in memory for this topic yet."

    response = client.responses.create(
        model=MODEL,
        instructions=(
            "You are a writing agent. Using only the provided research "
            "notes, write a short, polished paragraph for the end user."
        ),
        input=f"Research notes:\n{notes}\n\nWrite the final summary.",
    )
    return response.output_text


# ---- orchestration ----
if __name__ == "__main__":
    topic = "retrieval-augmented generation"

    researcher_agent(topic)              # writes to memory_store
    final_answer = writer_agent(topic)   # reads from memory_store

    print(final_answer)
What this demonstrates: the Researcher and Writer never talk to each other directly — they communicate only through memory_store, exactly like an episodic/external memory layer in a production system. Swap the dict for a real vector DB or SQL table and a read_memory() that does semantic search instead of exact lookup, and this becomes a genuine hybrid-retrieval multi-agent pipeline.
4.4

LangChain & LangGraph

The most widely used agent framework ecosystem — LangChain for composing LLM calls and tools, LangGraph for expressing an agent as an explicit stateful graph, and LangSmith for observing it all in production.

Framework

LangChain core: chains, runnables, and the LCEL interface

LangChain is a framework for composing LLM calls, prompts, tools, and parsers into reusable pipelines called chains, using a uniform interface called Runnables.

  • Chain — a sequence of steps (prompt → model → parser) that together perform one logical operation.
  • Runnable — the common interface (invoke, stream, batch) every component implements, so components snap together interchangeably.
  • LCEL (LangChain Expression Language) — the pipe syntax, e.g. prompt | model | parser, for declaratively composing Runnables into a chain.
Framework

Built-in tool integrations and custom tool creation

A "tool" in LangChain is a function the model can choose to call, described to the model by name, description, and argument schema.

LangChain ships integrations for common tools out of the box (web search, code execution, SQL, retrievers over vector stores). Custom tools are created by wrapping any Python function with a decorator (e.g. @tool) plus a docstring the model uses to decide when and how to call it — the docstring quality directly affects how reliably the model picks the right tool.

Framework

LangGraph: agent as a graph of nodes and edges

LangGraph models an agent explicitly as a directed graph: nodes are units of work (an LLM call, a tool call, a plain function), and edges define what runs next based on the current state.

This is a deliberate contrast to a purely linear chain: agent control flow is rarely a straight line (it loops, branches, and sometimes needs a human to step in), and a graph makes that control flow explicit and inspectable rather than buried in conditional Python.

Framework

State management in LangGraph — typed state schemas

Every LangGraph graph carries a single typed state object that every node reads from and writes updates to as it moves through the graph.

The schema (commonly a TypedDict or Pydantic model) declares exactly which fields exist — e.g. messages, retrieved_docs, iteration_count — giving type safety and a single source of truth for "what the agent currently knows," which doubles as its in-context/working memory for that run.

Framework

Cycles, branching, and human-in-the-loop nodes

  • Cycles — an edge can point back to an earlier node, enabling the Observe-Think-Act loop from 4.1 to run as a literal loop in the graph (e.g. "tool node → back to reasoning node") until a stop condition is met.
  • Branching — conditional edges route to different next nodes based on the current state (e.g. "if the model requested a tool, go to the tool node; otherwise end").
  • Human-in-the-loop nodes — a node that pauses graph execution and waits for human approval/input before continuing, used for the "add determinism / require confirmation" cases discussed under agent reliability in 4.1.
Tooling

LangSmith for tracing, debugging, and dataset management

LangSmith is the observability platform for LangChain/LangGraph applications: it records a full trace of every run — each node, prompt, tool call, and token used — for debugging and evaluation.

Beyond tracing individual runs, it supports building evaluation datasets from real or synthetic examples and running regression tests against them, which is how teams catch a prompt or graph change that quietly degrades agent behaviour before it reaches production.
