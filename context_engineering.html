01 — Foundation

What Is Context
Engineering?

Context engineering is the systematic practice of designing, selecting, and structuring the information that enters a language model's context window — the finite memory buffer that determines everything the model can reference when generating a response.

While prompt engineering focuses on how you instruct the model, context engineering focuses on what the model gets to see at all. It's the difference between writing a good question and building the right library before you even ask it.

This matters because every model — whether GPT-4o with 128K tokens, GPT-4.1 with 1M tokens, or Claude Opus with 200K tokens — has a hard boundary. Exceed it, and content is simply invisible to the model.

03 — Compression

Summarisation Strategies
for Long Histories

A 50-turn conversation at 2K tokens per turn = 100K tokens. That's the entire GPT-4o window gone. Summarisation is how you compress the past without losing what matters.

Rolling Summary

Maintain a running summary that gets updated every N turns. Old details fade, but key decisions and context persist.

Turn 1-5: Full messages
Turn 6-10: Summary of 1-5 + Full 6-10
Turn 11-15: Summary of 1-10 + Full 11-15
Reduces 100K → ~15K tokens

Sliding Window + Summary

Keep the last K turns verbatim for conversational nuance, plus a summary of everything before. Best of both worlds.

[Summary] Condensed turns 1-40
[Full] Turns 41-50 (verbatim)
[System] Current task prompt
Reduces 100K → ~25K tokens

Topic-Based Clustering

Group conversation into topics. Summarise completed topics fully, keep active topic verbatim. Discard off-topic digressions.

✓ Topic A: "Auth flow" (done, summarized)
→ Topic B: "Payment API" (active, full)
✗ Digression: "Lunch plans" (dropped)
Reduces 100K → ~20K tokens

Model-Specific Insight

GPT-4.1's 1M token window changes the math dramatically — you could fit ~500 pages of conversation verbatim. But even there, summarisation improves quality, not just capacity. Models perform better on focused, relevant context than on exhaustive dumps. Claude Opus excels at following complex instructions within its 200K window, making it especially sensitive to noise — clean context matters more than large context. GPT-4o Mini, with its smaller effective reasoning capacity, benefits most from aggressive summarisation.

04 — Anti-Pattern

Document Stuffing
Patterns & Limits

Document stuffing is the naive approach: dump everything into the context window and hope the model figures it out. It's the most common mistake in context engineering.

With GPT-4o's 128K window, people stuff all 128K. With GPT-4.1's 1M window, they stuff all 1M. The result? Degraded performance, slower inference, and higher cost.

Entire PDFs, codebases, or datasets dumped wholesale. Works for small docs, fails catastrophically at scale. Even GPT-4.1's 1M window won't save you — the model can't reason over 750K words effectively.

Same reference material re-inserted every turn. Common in chat interfaces that re-send the full context each API call. Wastes tokens and can confuse the model about what's current.

All context given equal weight — no signal about what's critical vs. supplementary. The model treats your 3-line instruction the same as a 50-page contract. Explicit priority markers ("IMPORTANT:", "CRITICAL:") help, but better structure solves this fundamentally.

Dynamic Context
Assembly

Instead of pre-loading a fixed context, dynamically select and compose relevant content at runtime based on the user's query, conversation state, and task requirements.

1

Semantic Routing

Classify the query type first, then load different context profiles. A code question gets API docs; a billing question gets account data. No single context serves all queries.

query_type = classify(user_msg)
context = load_profile(query_type)
# "code" → api_docs + type_defs
# "billing" → account + invoices
2

Relevance Thresholding

Set a minimum similarity score for RAG results. Chunks below 0.7 cosine similarity are excluded even if the window has room. Quality over coverage.

3

Recency-Weighted Selection

Boost retrieval scores for recently accessed or modified documents. If the user just edited a file, that file's chunks get priority over semantically similar but stale content.

4

Token Budget Allocation

Pre-allocate your token budget: 10% system, 5% instructions, 40% RAG, 30% history, 15% buffer. Each source gets capped, preventing any one component from dominating.

Contextual Caching Hints

With Anthropic's prompt caching, structure your context so static portions (system prompt, reference docs) can be cached between turns. This doesn't save tokens but cuts latency and cost by ~90% for cached portions.

Simple queries get minimal context; complex ones get the full budget. Estimate complexity first, then scale context accordingly.

06 — Cognitive Bias

Lost-in-the-Middle
Problem & Mitigation

Language models attend strongly to the beginning and end of context, but significantly less to information buried in the middle. This isn't a bug — it's an architectural property of transformer attention.

01

Bookend Placement

Put the most critical information at the very start (system prompt) and very end (just before the user's message). Never bury key instructions in the middle.

02

Repeat Critical Info

For instructions that must be followed, state them in both the system prompt and again near the end. Claude models are particularly responsive to repeated emphasis at boundaries.

03

Structured XML Tags

Use XML-style tags (<critical>, <context>, <instructions>) to create explicit boundaries. Claude models natively parse these; GPT models also benefit from clear delimiters.

04

Chunk Reordering

After RAG retrieval, reorder results: most relevant at start and end, least relevant in the middle. Don't just present them in retrieval-score order.

05

Reduce Middle Bulk

If you have 20 RAG chunks, don't stack them all in the middle. Summarize the middle 14 into a paragraph, keep the top 3 at the start and bottom 3 at the end in full.

06

Model-Specific Tuning

GPT-4.1 shows improved middle attention vs. GPT-4o. Claude Opus handles long middle sections better than Sonnet. GPT-5 is expected to further reduce this bias. Know your model.

07 — Efficiency

Context Compression
Techniques

Beyond summarisation, compression reshapes information to preserve meaning in fewer tokens. These techniques work at the character, token, and semantic levels.

Lexical Compression

2-5x

Remove stopwords, abbreviate, use telegraphic style. "The user wants to know about the payment processing system" → "User Q: payment processing system"

"Please ensure that you carefully review each of the following documents before providing your analysis"
"Review docs, analyze"

Structural Compression

3-8x

Convert prose to tables, lists, or JSON. Tabular data is far more token-efficient than natural language descriptions.

"The product costs $49. It has a 30-day refund policy. Support is available 24/7 via chat and email..."
{"price":49,"refund":"30d","support":["24/7 chat","email"]}

Semantic Compression

5-20x

Use a smaller model to extract only the propositions that are relevant to your task. Not a summary — a filtered extraction of meaning.

Original (5K tokens): Full contract text
Extracted (400 tokens): Liability cap: $1M. Term: 24mo. Termination: 30d notice. IP: work-for-hire. Governing law: Delaware.

Embedding-Based Pruning

Variable

Compare embeddings of context chunks against the query. Drop chunks below a threshold before they ever enter the prompt. This is pre-compression.

100 chunks × 500 tokens = 50K tokens
After pruning (cosine > 0.75): 12 chunks = 6K tokens
Savings: 44K tokens (88%)
