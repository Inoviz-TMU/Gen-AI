01 · Foundations

Measuring and optimising retrieval quality

A RAG system has two jobs: find the right information, then use it correctly. If either step fails, the final answer is wrong — even if the other step was perfect.

Most beginners only look at the final answer and ask "does this look right?" That's not enough, because a good-looking answer can hide a broken retriever, and a bad-looking answer can hide a retriever that actually did its job. To fix a RAG system, you need to measure each stage separately:

R

Retrieval quality

Did we pull the chunks that actually contain the answer? Measured independently of what the LLM later does with them.

G

Generation quality

Given the chunks we retrieved, did the LLM produce an answer that is faithful to them and actually useful to the user?

Why this matters: "optimising retrieval" almost always means changing chunk size, embedding model, top-k, or adding reranking — and you can only tell if a change helped if you have a number to compare before and after.
02 · Diagnosis

RAG failure modes

When a RAG answer is wrong, the bug is almost always in one of three places. Learning to tell them apart is the single most useful debugging skill in this session.

✕

Retrieval failure

The right information exists in the knowledge base, but the retriever never fetched it. The LLM never had a chance.

query: "refund window for electronics?"
retrieved: [shipping policy, returns FAQ]
verdict: MISSING — electronics clause never retrieved
!

Faithfulness failure

Retrieval worked — the right chunk was in context — but the LLM ignored it, mixed it up, or added facts that aren't there. This is hallucination despite good context.

context: "electronics: 15-day return"
answer: "You have 30 days to return electronics"
verdict: CONTRADICTS the retrieved context
?

Answer failure

Retrieval and faithfulness were both fine, but the answer still doesn't help the user — incomplete, badly scoped, or answering a different question than what was asked.

question: "how do I start a return?"
answer: "Electronics have a 15-day window."
verdict: true, but doesn't answer "how"
Debugging tip: always check the retrieved chunks first, before you judge the answer. If the right chunk isn't there, don't waste time tuning your prompt — go fix retrieval instead.
03 · Automated Evaluation

The RAGAS evaluation framework

RAGAS is an open-source library that scores your RAG pipeline automatically, using another LLM as a judge — so you don't have to hand-label every question yourself. It turns the three failure modes above into three numbers you can track over time.

Faithfulness
answer claims backed by context
Answer relevancy
answer actually addresses the question
Context recall
needed info was actually retrieved

Faithfulness

Catches faithfulness failures. RAGAS breaks the answer into individual claims, then checks each one against the retrieved context.

faithfulness = (claims supported by context) / (total claims in the answer)

Answer relevancy

Catches answer failures. RAGAS asks an LLM to reverse-engineer several questions that the given answer would be a good response to, embeds them, and compares them against the embedding of the original question. Off-topic or incomplete answers score low.

answer relevancy = average similarity( original question, questions reconstructed from the answer )

Context recall

Catches retrieval failures. Using a reference ("ground truth") answer, RAGAS checks whether everything needed to produce that answer was actually present somewhere in the retrieved chunks.

context recall = (ground-truth statements attributable to retrieved context) / (total ground-truth statements)
Python — running all three metrics
# you already have: questions, generated_answers, retrieved_contexts, reference_answers
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_recall
from datasets import Dataset

data = {
    "question": questions,
    "answer": generated_answers,
    "contexts": retrieved_contexts,     # list[list[str]]
    "ground_truth": reference_answers,
}

result = evaluate(
    Dataset.from_dict(data),
    metrics=[faithfulness, answer_relevancy, context_recall],
)
print(result)  # {'faithfulness': 0.93, 'answer_relevancy': 0.88, 'context_recall': 0.71}
Reading the example above: faithfulness and relevancy are strong — generation is trustworthy. Context recall at 0.71 is the weak link: the retriever is missing information the answer needed. That's a signal to revisit chunking or add reranking, not to touch the prompt.
04 · Speed & Cost

Semantic caching with Redis / Momento

Every RAG query normally costs one embedding call, one vector search, and one LLM generation call. If many users ask the same thing in different words, you're paying for the full pipeline every single time. Semantic caching skips it.

Instead of caching by the exact text of a query, you cache by meaning: store the embedding of every question you've answered, and before running the full pipeline, check whether a sufficiently similar question has already been answered.

Architecture
1 · User query"How do I reset my password?"
→
2 · Embed queryturn text into a vector
→
3 · Vector lookupsearch Redis / Momento cache for a similar past query
→
HIT (similar enough)return cached answer — no LLM call
MISS (nothing close)run full retrieval + generation, then store the result

Redis (with the RediSearch / vector module) and Momento Vector Index both work the same way here: they store a vector alongside a payload (the cached answer) and let you run a nearest-neighbour search with a similarity threshold at read time.

TTL strategy

TTL (time-to-live) controls how long a cached answer is trusted before it's treated as stale and evicted. Match it to how fast the underlying knowledge changes:

Content typeSuggested TTLWhy
Static docs (policies, product specs)Days – weeksRarely changes, safe to keep long
Semi-dynamic (FAQs, pricing tiers)HoursChanges occasionally; balance freshness vs. hit rate
Volatile (stock, live status, news)Seconds – minutesStale answers here are actively misleading
Python — cache check & store (Redis)
import redis, numpy as np
r = redis.Redis(host="localhost", port=6379)

def check_cache(query_vec, threshold=0.92):
    # KNN search against previously cached query embeddings
    hit = r.ft("query_cache").search(
        "*=>[KNN 1 @embedding $vec AS score]",
        query_params={"vec": np.array(query_vec, dtype=np.float32).tobytes()},
    )
    if hit.docs and float(hit.docs[0].score) >= threshold:
        return hit.docs[0].answer          # cache HIT
    return None                            # cache MISS

def store_cache(query_vec, answer, ttl_seconds=3600):
    key = f"cache:{uuid4()}"
    r.hset(key, mapping={"embedding": np.array(query_vec, dtype=np.float32).tobytes(),
                          "answer": answer})
    r.expire(key, ttl_seconds)   # entry auto-evicts after TTL
05 · Trade-offs

Exact-match vs. semantic similarity caching

Both are valid — they just catch different kinds of repeated questions.

=

Exact-match caching

Hash the literal query string and look it up. O(1), cheap, no embedding call needed.

Only hits when the incoming text is character-for-character identical (or normalised — lowercased, whitespace-trimmed).

≈

Semantic caching

Embed the query and search for a meaning-similar past query above a similarity threshold.

Hits on paraphrases, but costs an embedding call + vector search, and a threshold set too loose can return a wrong-but-similar answer.

Cached question: "What is your return policy?" — see which incoming queries would hit the cache.

Incoming querySimilarityExact-match resultSemantic result
"What is your return policy?"1.00 HIT HIT
"what's your return policy?"0.99 MISS (punctuation differs) HIT
"How do I return an item?"0.94 MISS HIT
"What is your refund policy?"0.88 MISS MISS (below 0.92 threshold — different concept)
Rule of thumb: use exact-match as a free first layer (it's nearly free), then fall back to semantic caching for the long tail of paraphrased questions. Most production systems run both.
06 · Freshness

Cache invalidation for dynamic knowledge bases

A cached answer is only useful if it's still true. When your source documents change, stale cache entries need to be found and removed — this is what "invalidation" means.

T

TTL expiry

Every entry auto-expires after a fixed time. Simple to implement, but coarse — you either evict too early (losing hit rate) or too late (serving stale answers).

E

Event-driven invalidation

When a source document changes, publish an event that purges only the cache entries built from that document. Precise, but requires wiring your document pipeline to your cache.

V

Version tagging

Tag each cache entry with a hash/version of the source document it came from. On read, compare versions — a mismatch means "stale, treat as a miss."

In practice, most teams combine all three: TTL as a safety net (nothing lives forever), event-driven purges for known updates, and version tags to catch anything the event system missed.

07 · Putting It Together

End-to-end RAG benchmark

A benchmark is what turns "I think it's better" into "here's the number that proves it." Build it once, then re-run it every time you change chunking, the embedding model, top-k, or the prompt.

Build a golden dataset

A set of realistic questions, each with a reference ("ground truth") answer and, ideally, the source chunks that should be retrieved. 30–100 questions is a reasonable starting point.

Run the pipeline and log everything

For each question, record the cache hit/miss, the retrieved contexts, the generated answer, and the latency. You need all of this to diagnose failures later.

Compute the metrics

Run RAGAS over the logged results for faithfulness, answer relevancy, and context recall. Separately compute cache hit rate and latency percentiles.

Interpret and iterate

Low context recall → fix retrieval. Low faithfulness → fix the prompt or generation. Low hit rate → loosen the similarity threshold or widen TTLs.

Python — benchmark loop
import time

logged = []
for item in golden_dataset:
    start = time.time()
    q_vec = embed(item["question"])
    cached = check_cache(q_vec)

    if cached:
        answer, contexts, hit = cached, None, True
    else:
        contexts = retrieve(item["question"])
        answer = generate(item["question"], contexts)
        store_cache(q_vec, answer)
        hit = False

    logged.append({
        "question": item["question"],
        "answer": answer,
        "contexts": contexts,
        "ground_truth": item["reference_answer"],
        "cache_hit": hit,
        "latency": time.time() - start,
    })

The habit to build: re-run this exact benchmark after every pipeline change. If a number doesn't move — or moves the wrong way — you just saved yourself from shipping a "fix" that wasn't one.
